OOP和FP

类和hook延伸到的OOP和FP

js有两股力量：一类是面向对象编程（OOP，vue，mobx），一类是函数式编程（FP，hook和redux）

而函数式编程的核心概念：

数据不可变： 它要求你所有的数据都是不可变的，这意味着如果你想修改一个对象，那你应该创建一个新的对象用来修改，而不是修改已有的对象。

无状态： 主要是强调对于一个函数，不管你何时运行，它都应该像第一次运行一样，给定相同的输入，给出相同的输出，完全不依赖外部状态的变化。

管道一样：结果经过一个一个管道（函数），传递下去，最后返回给用户。

react官方，是全面拥抱hook，你不想用，但也得陪跑这种感觉。

对hook的一些看法：

Hooks提出的逻辑分离，并没有超越容器组件与展示组件所表达的设计思想。而在写法上，也没有简便到超越装饰器+HOC。从代码组织上看，最起码HOC可以降低对React组件的入侵程度，而Hooks要求写在组件或自定义Hooks内部。

ts是针对js不能胜任大型应用而开发的一门语言，两个特性：类型系统和面向对象。但是hook和redux与ts的匹配不是十分贴切，主要是缺少了面向对象，在代码体量增长，模块复杂的时候，面向对象有很好的解决方案

vue/mobx的响应原理天然适合对接OOP模型，react响应原理天然适合对接FP模型。

选择proxy响应式+OOP模型，还是选择react响应式+FP模型。小项目用vue快。大项目用react，数据流向清晰。

react的核心理念就是FP，一个状态生成一个UI，自己维护着状态机，

第一，vue之前的option-based api，和react之前的class api，都不是面向对象的写法。面向对象，如今有几种主流的理解，不管是c++ java系的封装继承多态，还是erlang式的messaging，都和vue react关系不大。事实上两大框架都避免继承，也用不到多态。

第二，vue和react现在的hook/composable风格api，也不是真正的函数式编程，顶多可以说react hooks稍微接近一点函数式编程而已，但也不能说react用js搞定了典型函数式编程的各种主要特性，毕竟js语言本身的限制在那里。vue就更远了，vue的根基是对可变数据的观察跟踪，而函数式编程的基础就是不可变数据。

总之，前端框架，也就是ui的数据联动和组件化这个领域，是一个挺独特的编程领域，这些年发展出来了很多自己独特的范式，不宜简单套用传统的oop fp概念。顶多可以说，它们的组件概念借鉴了oop的封装概念，但抛弃了oop的继承和多态；react又额外借鉴了一些fp中的风格。

数据可变 数据不可变
class 甲风格 典型OOP，适合对接vue、mobx；通过immer对接react不成熟 丙风格 非典型FP，class外表下所有方法都是纯函数，适合对接react，包括class和hooks
函数 丁风格 非典型OOP，相当于把class的方法外置了，适合对接vue、mobx也可以通过immer对接react 乙风格 典型FP，适合对接react，包括class和hooks

FP和OOP都是一种编程思想，而思想是抽象的，因为是抽象的事物，所以每个人认知都并不完全一致，有的极致纯粹，有的混沌模糊。

OOP的核心是对世界的抽象，即万事万物皆为对象，以抽象描述事物，用事物构成程序，是编程人员对这个世界的主观感性认知，是一种符合人类对事物认知的表达方式，编程范式具有抽象、封装、继承、多态的特点。

FP的核心是函数，脱胎于数学中的定义域（x）、法则(f)、值域(y)，对应了编程函数的输入(I)、运算规则(Expression)、输出(O),是all in function的表象，客观理性、纯粹计算的思考方式，编程范式有不修改状态，引用透明、无副作用、只用表达式不用语句的特点。

这两种编程思想，被具化到了语言当中，即语言特性是否支持对这两种思想的表达，而JavaScript恰恰以灵活著称，两个编程方式都“部分”支持。

也由此产生了关于编程思想的信仰之争，有了争议也就有了对错是非的探讨，但抛开实际问题去谈绝对的对与错都是片面的，A说他的项目里OOP可以hold住一切，B说他的项目里只允许FP的方式，是完全无效的对话。

函数式编程给自己挖的坑

函数式编程一直以来都是React的核心理念，所以用函数式方式写UI是自然的，但是真实世界跟数学不一样，不是只有一个输入产生一个输出，同时还要有其他的副作用，比如动画、通讯等等，这些不是纯函数能够实现的，为此需要想办法在纯函数中加入这些副作用才行。

Facebook又一次给自己挖了坑，他们将副作用写成useXXX的形式，其实是把这些副作用都加入到一个全局处理队列中执行了， 而且更麻烦的是这个执行队列是线性的！意味着除非把前面的任务处理完毕，否则后面的任务只能干等！而为了保证逻辑完整性，还不能使用并行方式处理！这么大的坑怎么等发布了才想到！这就导致了Facebook需要进一步给自己挖坑，不，是自救，所以又搞出来分时执行，也就是一个任务计算不完，不管你了，你先暂停，我先去渲染一下UI再来搞你。

表面看起来React使用的技术越来越高级，其实是它自己给自己挖的坑越来越大，真不知道后面它还能挖出什么样的坑出来。

而反观Vue的复合函数似乎就没有React这样的尴尬，原理还没有去深入了解，但是按照尤大的开发态度肯定是解决了React的坑才会选择使用函数式编程方式的，否则这么大的一个性能坑在这里一个人是肯定搞不定的。

从题主的描述，很高兴题主是一个有独立思考的人，其实说句不当说的话，现在前端大部分人都是人云亦云之辈，很少有自己的独立判断，而且跟风之气不但没有随时间淡化，反而有愈演愈烈之势。

不得不说，hooks 的提出对前端开发是一件非常有开创性思维的事，react 的虚拟 dom + jsx 亦是如此，函数式编程也喧嚣日上，但要说这就是前端最优解，我觉得倒是可以斟酌之事。

每一种方式都是一种解决问题的思路和探索，现在很多人容易非此即彼。一说到函数式，就说面向对象一无是处。一说到 react，就说 vue 或 angular 都是垃圾。每当有听到看到如此言论时，大可不必当真。一部分是工作时间短，还缺少一些实践经验，缺乏判断，可以理解。另一部分属于思维固化或随波逐流之辈，就更不必争论了。

框架、语言都是工具。就个人的一点浅见，如果作为一个技术负责人，当然要考虑团队成员的意见，还有招聘成本，培养成本等，这些均为可作为技术选型的参考。但如果仅因此就拾糟粕而弃珠玉，那你的作用在哪里？这时，我觉得更需要一个经验丰富，有远见的人来决策。记住，团队成员只对自己负责，而你要为团队负责，团队成员做得不开心，换一个坑，而你作为一个负责人，换坑就没那么简单，在大多数情况下，可能你会在这家公司工作多年。
