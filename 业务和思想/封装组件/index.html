封装组件

​

​ 首先确定是受控组件还是非受控组件，组件是否有自身的状态，如果是受控组件，没有自己的状态，内部展示的UI都是外部传入，那么直接props传入组件内部，组件内部维护一些默认值，该组件相当于一个layout，我们维护好props，该组件即可完成。

​ 下面来说说有状态的组件，先说自己的状态state，自己的的state一定会被在某个地方消费，需要确定这个state是在哪消费的（在哪发送的网络请求），如果说是子组件内部，自己消费，那么直接内部发生动作皆可，如果是外界需要，比如说父组件，那么子组件在触发某个动作的时候，就要讲state emit出去，父组件响应拿到state。

其次如果是一些form表单，我们完全可以把一个form表单封装成一个API，运用命令模式，表单内部自己维护state，和一些getState的方法，由父组件拿到form的ref来调用form的的方法。

​ 外部传入的props，比较常见的就是，父组件传入不同的id，子组件展示不同的内容，子组件内部使用watch监听id的变化，子组件监听id发送网络请求，切换展示UI。react的话将id放在useEffect的依赖项中，进行网络处理。这只是一个场景，还是要具体场景，具体分析。

​ 除了状态，还可能会有回调或者一些手势逻辑操作等等，这一部分需要具体分析。

​ 对于一些文字展示，比如说labelText，外部正常传入，内部正常布局即可，同时可能还会有一些样式变量的传入，比如说外部传入数量，决定内部的珊格的列数，我们可以用css变量来做。

​ 封装的组件可能是相同的样式，比如说dialog，一般来说用fixed布局和teleport（传送门）来做，封装成一个高阶组件，然后再通过props render这种方法来渲染组件。

​ 高阶组件，也是封装组件时候常用的方法，根本原理是利用函数柯西化，将一些变量内部，做一些类似于中间的处理，劫持props，逻辑的复用，比如一些组件共享的数据context，我们可以用高阶组件做一个中转站， 在高阶组件的内部使用一些共享数据的逻辑标签，把自己想用的组件包裹进去，然后给组件传值即可。鉴权，缓存声明周期。主要思想还是逻辑复用。

​ 写业务代码，整体上来说是一个熵增的过程，不管一开始多么优美的代码，到后面都会变成 山，因此如何开始和维护就显得尤为重要。需要有设计模式的运用。多看、多写、多想，把代码写好，维护好，服务好业务，才是最重要的。

如何封装组件、粒度和边界如何决定

（单一性：单一性要求一个组件具有高内聚，低耦合的特征，它只负责一件事情，不要耦合一些没必要的逻辑，并且尽量不要和其他组件有过于多的双向交互和互相依赖关系。单一性并不代表着不可以引用其他组件，当前组件可能是外层的容器组件，里面包含一些子组件，这样的设计是没问题的。）

（复用性：在设计组件的时候，一定要考虑组件的复用性或者说是通用性。这是指，当组件封装好后，可以在类似的使用场景中直接调用。这要求我们在设计组件的时候，考虑组件功能的通用性，以及考虑组件入参的合理性。）

（业务组件：业务组件侧重于数据和业务的逻辑处理，其中数据一般通过接口获取。目前本团队维护的业务组件库，可以使开发人员即来即用，组件内部有完善的功能和接口数据处理，将组件引入到项目后可直接实现对应功能。）

​ （UI 组件：UI 组件一般也可以称为基础组件，它们经常在多个地方被复用，且不耦合任何的业务功能，例如：AntD 组件库。UI 组件侧重于页面展示效果，大部分 UI 组件具有原子性，一些复杂的 UI 组件可以由基本的 UI 组件构成。一般情况下组件内部的数据来源于父组件传递过来的 props。）

（原子组件 ==> 分子组件 ==> 生物组件 ==> 模板组件 ==> 页面）
