Cookie

​ cookie是存储在浏览器上的一小段数据，用来记录某些当页面关闭或者刷新后仍然需要记录的信息,服务器生成，并且不需要浏览器做任何事件，随请求每次提交，设置过期时间的时候，存于硬盘，永久存储，一般浏览器存储cookie最大容量为4kb，没有设置的时候，会在内存之中。服务器会把set-cookie放在响应头里，浏览器会自动读取set-cookie，并且保存到请求头中。下一次请求会自动携带cookie。

cookie中的其他属性：

  1. Expires：Expires 用于设置 Cookie 的过期时间。
  2. Max-Age：用于设置在 Cookie 失效之前需要经过的秒数。
  3. Domain：指定了 Cookie 可以送达的主机名。
  4. Secure（sirQ若）：标记为 Secure的 Cookie 只应通过被HTTPS协议加密过的请求发送给服务端。使用 HTTPS 安全协议，可以保护 Cookie 在浏览器和 Web服务器间的传输过程中不被窃取和篡改。
  5. HTTPOnly：可以防止客户端脚本通过document.cookie等方式访问 Cookie，有助于避免 XSS 攻击。
  6. SameSite：可以让 Cookie在跨站请求时不会被发送，从而可以阻止跨站请求伪造攻击（CSRF）。

cookie的设置

​ 当服务器收到 HTTP 请求时，服务器可以在响应头里面添加一个 Set-Cookie 选项。浏览器收到响应后通常会保存下 Cookie，之后对该服务器每一次请求中都通过 Cookie 请求头部将 Cookie 信息发送给服务器。另外，Cookie 的过期时间、域、路径、有效期、适用站点都可以根据需要来指定。

cookie的生命周期

  • 会话期 Cookie 是最简单的 Cookie：浏览器关闭之后它会被自动删除，也就是说它仅在会话期内有效。会话期 Cookie 不需要指定过期时间（Expires）或者有效期（Max-Age）。需要注意的是，有些浏览器提供了会话恢复功能，这种情况下即使关闭了浏览器，会话期 Cookie 也会被保留下来，就好像浏览器从来没有关闭一样，这会导致 Cookie 的生命周期无限期延长。
  • 持久性 Cookie 的生命周期取决于过期时间（Expires）或有效期（Max-Age）指定的一段时间。

提示：当 Cookie 的过期时间被设定时，设定的日期和时间只与客户端相关，而不是服务端。

cookie存在的问题：

  1. 安全性：cookie中的所有数据在客户端就可以被修改，数据非常容易被伪造，所以对于用户信息来讲，应该将与登录相关的重要信息存储在服务端的session中。
  2. cookie虽然不能跨域携带，但是却可以跨站点共享，你向同一服务器发送请求时，会带上浏览器保存的对于那个服务器的所有cookie，而不管你从哪个网站发起的请求。
  3. 性能：如果cookie中数据字段太多的话，不仅影响传输效率，还容易泄密，所以将重要的信息存放在服务端的session中。

引出的session：

​ 存储于服务端，浏览器发送请求给服务器时，服务器生成session，并将用户的重要信息存储于session中，并将session-id存储于cookie中，响应式的时候，携带着set-cookite的相应头，给浏览器，浏览器种下cookie。在下一次浏览器访问服务器时，cookie中携带session-id，可以和服务器的session相对应。即记录了页面的信息，又保护了用户的重要信息。

session存在的问题：

session都存在一个服务器上，当负载均衡器把用户请求转发到另一个服务器上，由于此server没有该站点的cookie，用户就要重新登录。

​ 问：如果是服务器集群，就要求 session 数据共享，每台服务器都能够读取 session。A 网站和 B 网站是同一家公司的关联服务。现在要求，用户只要在其中一个网站登录，再访问另一个网站就会自动登录，请问怎么实现？

  1. 负载均衡指向一个服务器（请求过大，如果服务器宕机，丢失所有数据）
  2. 集群服务器进行同步cookie操作，过于消耗内存。
  3. 单独的数据持久化服务器，服务器收到请求都向持久层请求。缺点，格外编码，另外，持久层万一挂了，就会单点失败。（现在的解决方案）

另外的一个办法就是JWT（跨段身份验证解决方案）

概述

header（声明方式）和payload（用户不敏感信息）经过base64编码，得到一串编码，在经服务端私钥加密算法，生成签名（token），返回给浏览器，浏览器下次请求携带token，server下次接收到token的时候，公约解密，在base64反编码，得到用户信息。

JWT的思想是去中心化，数据保存在客户端，不在依赖服务端。

token的优势

  1. 可做跨端，不再依赖cookie，只有客户端可以存储信息，token就可以使用。
  2. 单点登录友好，因为不在依赖服务器，
  3. 无状态，token 自身包含了身份验证所需要的所有信息，使得我们的服务器不需要存储 Session 信息，这显然增加了系统的可用性和伸缩性，大大减轻了服务端的压力。
  4. 防止了CSRF攻击，就算引诱到第三方网站，token 并不会自动被浏览器放入请求头，当其他攻击者像服务器发送请求时，是不会携带token 的，后端服务器没有获取到token 则不会对请求进行放行，保证了用户数据的安全性。

token存放位置：

  1. url拼接；
  2. header中（放在localStorage中，axios请求拦截器中添加自定义的请求头中 config.headers.Authorization =Bearer ${token}，但是如果网站容易收到XSS攻击，存储在localStorage将不会安全，为了防止XSS攻击，避开所有不可信数据，
  3. cookie中（不可跨域）但是有cros的问题，可能会诱导用户执行错误，可以执行httponly防止js读取，不符合
  4. 参考链接：token在哪里，自定义header的token，依赖前端，cookie token 依赖后端。
  5. token过期，无感刷新，针对于不同的市场有不一样的操作。

双token保证用户活跃，token续签

token存在的问题：Token用于身份认证时，如果有效期设置太长，泄露了会不安全。如果设置太短，用户频繁的重新登陆

用户登录返回两个 token ：第一个是 acessToken ，它的过期时间 token 本身的过期时间比如半个小时，另外一个是 refreshToken 它的过期时间更长一点比如为1天。客户端登录后，将 accessToken和refreshToken 保存在本地，每次访问将 accessToken 传给服务端。服务端校验 accessToken 的有效性，如果过期的话，就将 refreshToken 传给服务端。如果有效，服务端就生成新的 accessToken 给客户端。否则，客户端就重新登录即可。

该方案的不足是：(1).需要客户端来配合；(2).用户注销的时候需要同时保证两个 token 都无效；(3).重新请求获取 token 的过程中会有短暂 token 不可用的情况（可以通过在客户端设置定时器，当accessToken 快过期的时候，提前去通过 refreshToken 获取新的accessToken）。

cookie、sessionStorage、localStorage之间的区别

基础说明

  1. 存储大小：一般浏览器存储cookie最大容量为4kb，很多浏览器都限制一个站点最多保存20个cookie。sessionStorage和localStorage虽然也有存储大小的限制，但是要比cookie大得多。

  2. 存储有效期：

    1. cookie保存的数据在过期时间之前一直有效，即使关闭浏览器
    2. sessionStorage保存的数据用于浏览器的当前窗口或者一次会话，当会话结束（通常是窗口关闭），数据被清空，需要注意刷新或者强制清空缓存都不能让sessionStorage失效，不支持跨标签页。
    3. localStorage保存的数据长期存在，因为localStorage的数据是写入磁盘中，每次读取数据时，实际上是从硬盘驱动器上读取这些字节。

  3. cookie参与网络传输，后两者只是存储方式，不参与网络传输。

对两个Storage同源策略的说明
  1. 同一浏览器的相同域名和端口的不同页面间（同域名）可以共享相同的 localStorage
  2. 但是不同页面间无法共享sessionStorage的信息。这里需要注意的是，页面仅指顶级窗口，如果一个页面包含多个iframe且他们属于同源页面，那么他们之间是可以共享sessionStorage的。
  3. sessionStorage在同一标签页可以共享，包括a链接（同一标签页跳转），
  1. 存储需要转译，字符串转js对象

问：已经有了cookie了，还需要localStoreage和sessionStorage呢？

答：因为在请求的时候，request header中会携带cookie信息，每次请求的时候都会有全量的cookie信息进行上传，就会浪费带宽，所以h5提出local Storeage这个新特性，解决这个问题。sessionStorage的意义：会话存储

  1. indexedDB存储方案：正如大多数的 web 储存解决方案一样，IndexedDB 也遵守同源策略。因此当你在某个域名下操作储存数据的时候，你不能操作其他域名下的数据。
  2. localStorage存储可以做性能监控，在一个页面跳转到另一个页面，在离开的时候埋下钩子，进入新页面的时候标记时间戳，相减页面跳转的时间。
  3. localStroage可以监听回调，当一个页面改变了Stroage，在同源别的页面可以监听到，添加回调。 Window.addEventlistener(“storage”,function (e){alert(e.newValue)})。监听本页面可以充血setItem方法，中途抛出自定义事件。
