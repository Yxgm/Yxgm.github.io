从更高层的设计思想看两个框架

从更高层的设计思想看两个框架

1.原生和抽象

react更加接近原生开发（手动挡），而vue更处于一个抽象概念（自动挡）

2.编译和运行的占比差异

vue，注重编译阶段

  1. 大致的原理就是把 template 解析成一个函数(render函数)，缺点也很明显，就是每次数据变化，模板内部要全部重新渲染。只会编译一次，也就是在构建的时候。

  2. vue3性能提升：预编译时做了静态节点标记，事件缓存，v-if和 v-for 内部通过 block+ 数组的方式维护动态元素等操作，（将不变的节点闭包放在render外，不用在走一遍h函数）由于预编译的这些优化，后期在数据更新的时候可以直接复用，更快速的生成虚拟DOM，而且还可以在diff的时候，永远跳过静态节点。提高diff性能。

react，注重运行阶段（几乎都在运行时）

  1. 数据发生变化后，生成一个新的尽可能复用的fiber，并且通过 diff 算法得出最小的操作行为，全部都是在运行时来做的

  2. 纯js写法，方法很灵活，但是使它在编译时很难做太多的事情，所以说react的历代更新都在解决自己运行时的优化，推出了时间切片和任务分片概念，榨干浏览器性能。

3.更细化的设计维度

vue可变数据，基于数据的拦截操作，让我们可以实现修改js对象的同时，就修改view层，这就是vue给我们提供的mvvm的黑盒子，让我们全身心注重于业务。

react不可变数据，基于Immutable思想，每次都是新的state，diff然后在算出差异，在commit渲染视图。心智模型较重，对与无用渲染的子组件的细致负担，需要自己来承受。

后期作出的权衡：

最开始的vue监听器很多，并不是以组件为粒度，后期Vue 选择了权衡，以组件作为颗粒度，组件层面用响应式通知，组件内部通过 dom diff 计算 ，既控制了应用内部 Watcher 的数量，也控制了 dom diff 的量级。

而react 推出fiber，时间切片，任务分片，任务调度等思想架构，让react刚开始雪崩的性能也好转起来。

时间切片:

vue3 通过静态标记 + 响应式 + 虚拟 dom 的方式，控制了 diff 的颗粒度，让 diff 的时间不会超过 16ms，但是 React 自上而下的 diff 过程，项目大了之后，一旦 diff 的时间超过 16.6ms，就会造成卡顿，

对此 React 交出的解决方案就是时间切片，就是把 diff 的任务按照元素拆开，利用浏览器的空闲时间去算 diff，React 把各种优化的策略都留给了开发者，Vue 则是帮开发者做了很多优化的工作

4. 组合优于集成

这个思想设计模式里面就有定论，也是现在 hooks 和 composition 大行其道的原因，代码写出来也会更加易于维护，这个图可以很好地体现出可维护性上的变化

5. 跨端

刚才我讲了 Svelte(扫物体) 可以做到直接编译成 JavaScript，性能接近原生，这么优秀的思想，为什么 Vue 还要保留虚拟 dom 这个额外的 runtime 损耗呢，我觉得比较重要的一个答案就是跨端

虚拟 dom 除了可以用来计算最小的 diff 之外，另外一个重要的功能就是可以用 JavaScript 的对象来去描述一个 dom，这就是一个普通的对象，在跨端领域意义重大，视图层返回的是一个对象，渲染层可以调用不同平台的渲染 api 去绘制即可
