redux的缺点

  1. 默认只支持同步处理，连作者创造的redux-thunk都是独立于redux之外的一个包，当然，这样是让使用者自行选择异步处理方法，但是选择太多，又没有各项都见长的解法，所以使用者也抱怨很多；连最常见的异步场景都要单独配个中间件。还有就是基本上正规项目用的朋友应该都做过二次封装

  2. 啰嗦（verbose），为了一个功能又要写reducer又要写action，还要写一个文件定义actionType，显得很麻烦，当然啰嗦就是为了让一切清晰明确；

  3. store设计，过于扁平化不好管理，但是树结构过深修改麻烦，需要一定的经验。

  4. state修改不方便。每次修改都需要返回引用不同的对象，为了书写方便，不得不借助immutable这样的工具，让调试不再直接可读(会有大量toJS的滥用导致的性能问题)

  5. state很容易滥用。对程序而言，数据结构要比代码重要，但重视数据结构的人真心不多，这会导致大部分的状态都会被放到state中，大部分组件都在connect store。

  6. 组件与store耦合度高，组件复用困难

  7. 性能问题。每次action会触发所有reducer的处理；每次state变更会导致所有connect的组件重新执行mapStateToProps，这个函数写的不太理想或者组件没有对props的修改进行渲染判定会导致大量的组件重新render

  8. 发布订阅的模式，比较浪费内存

  9. 使用 Redux 的话，明明很简单的一个应用改动，都必须要碰 n 个文件才能完成。代码每一行都很明白，但要宏观了解整个应用的状态机必须 n 个文件交叉看。

  10. 模板代码太多，使用不方便，属性要一个一个 pick，对 ts 也不友好，状态的修改重度依赖 immutable，计算属性要依赖 reselect，还有魔法字符串等一系列问题，心智负担大，用起来很麻烦容易出错，开发效率低下

  11. 触发更新的效率也比较差，connect 的组件的 listener 必须一个一个遍历，再靠浅比较去拦截不必要的更新，在大型应用里面无疑是灾难

  12. store 的推荐数据结构是 json object，这对于我们的业务来说也不太合适，我们的数据结构是图状的数据结构，互相有复杂的关联关系，比如父子兄弟层级、环状结构、链式结构、多对多等，比较偏向于后端数据模型，适合用面向对象来描述模型，描述切面，需要多实例隔离，显然用 json 或者 normalizr 强行做只会增加复杂度，和已有的代码也完全无法小成本适配

  13. 基于 snapshot 的 time travel 内存占用高，性能差，并且没有配套的暂停、重启、切换、清空、事务等机制，还有一大堆复杂的问题都没有考虑，更多是面向简单 web dev tool 场景

  14. 无法做关联更新，每一次的所有变更必须由用户触发，既是优点又是缺点，优点是没有魔法，缺点是在复杂业务里面心智负担更大，更容易写出 bug，比如我们业务里面一个门板需要自动根据柜子的大小做适配，如果用 redux 就得让每一个影响柜子大小的变更都记住要多加一行适配门板的更新操作，这显然是不可取的（我们需要避免魔法，但在一些稳定关联更新的业务逻辑里面我们需要正确的使用魔法）
