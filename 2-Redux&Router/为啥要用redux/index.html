为什么要用redux

回答一

​ 全局变量本质是一个单例模式，redux在单例的基础上加了发布订阅机制，采用reducer约束修改，再加上middleware机制保证拓展性。

​ React 的核心价值观是：状态决定渲染结果，因此只有状态改变了才需要重新渲染。如果你用一个非状态的全局变量来保存状态，那任何一个 React 组件都不会因为这个全局变量的人改变而重新渲染。

​ 在没有 hook 之前，Redux 通过 HOC 来解决这个问题。Redux 状态变化会触发 HOC 里的属性变化，因而触发被 HOC 包裹的原组件重新渲染。在有了 hook 之后，React 观察 hook 是否触发状态变化，Redux 通过 hook 来通知 React 状态变化。

​ 如果你想要让你的全局变量发生变化时能通知 React，那么你需要把上述 Redux 做的事情再做一遍。如果这个全局变量内的任何变化都通知每一个 React 组件重新渲染，那就会造成过度重复渲染。你希望一个变化只通知关心这个变化的相关组件，你在实现了这个优化之后你就重新发明了 Redux selector。

​ Redux 确实有很多简单场景用不到的「设计模式」，例如要有 action 然后还要有 action creator 这种，我把它们称之为过度设计。把这些砍掉后，你就得到了一个简单一些的带 selector 的状态机

回答二

​ 要知道redux重点在于数据流而不仅仅是数据源，它是一个动态的概念，如果仅仅是对于redux store而言，这个问题可以成立，答案是“可”，一个变量，要挂成全局变量还不简单。

​ 但是对于redux来说，毫无关系。不过抛开问题本身redux的核心在于一套让数据流动起来的模型，仅仅替换全局化数据源是无关紧要的，还需要能够触发更新，像window.store.xxx=xxxx这样子是触发不了UI更新的，因此要么像react搞个setState通知一下入个更新队列，vdomdiff就能高效保证到我们的UI更新了。

​ redux关键在于数据流动，而不在于数据源头，所以无关全局变量与否。

什么是数据的不可变性immutable

​ 对于setState，还是redex中的reducer，都是默认使用的全等比较，true，组件不渲染，false，组件渲染。

​ 如果数据的更新需要带来视图的重新渲染，那么一定要返回新的对象。假如我们对immutable毁约，破坏不可变性，在原来的对象上修改，那么也就导致组件无法刷新，后果可想而知。

​ 但是每次都要返回新的对象，在平时的开发中，一般都是使用解构到新的对象。如果state深层嵌套的时候，我们就又要使用深拷贝的方法，性能受很大影响。

​ 所以说immutable的api出现了，返回新的对象，并且尽可能复用对象的属性。依次来改善性能。相比于深拷贝，immutable.js是使用了structural sharing的技术。当我们对一个immutable对象进行操作的时候，immutable.js基于hash map trees和vector map tries，只拷贝当前修改的节点以及它的祖先节点，其他保持不变，不同的对象之间可以共享相同的部分。

浅比较：对于长得一样的对象，也会返回true，当时只是针对于第一层会做这样的判断，但是还有嵌套的引用的时候并且长的一样，将会返回false，比较结果为不相等，组件renderer。
