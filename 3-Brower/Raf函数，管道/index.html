2022-03-05 | 浏览器

requestAnimationFrame，MessageChannel

顶层理解：

​ 浏览器中的一帧：对于一帧来说，只是一个期望的时间，如果有宏任务不断地进入的话，浏览器渲染完一帧之后，可能是10ms的执行时间，都有可能，对于这一帧画面已经完成渲染。随之开启下一个宏任务，我们在做一些计算，这都是可以的。因为主要的渲染任务都已经执行完了，js线程和GUI线程都是空闲的，我们在开一个宏任务，没有问题。所以说react在理想一帧的中，将浏览器的剩余时间让宏任务占满。在剩余的一帧中，执行work。

requestAnimationFrame

执行一个动画，并且要求浏览器在下次重绘之前调用指定的回调函数更新动画。该方法需要传入一个回调函数作为参数，该回调函数会在浏览器下一次重绘之前执行

一个eventLoop的完整过程是包含这浏览器的渲染过程的，raf的执行会在一个eventLoop中的微任务结束后，下一个eventloop开始前去执行，准确的说就是本次一帧的渲染之前执行

  • raf任务队列被执行的时候，会将其此刻队列中所有的任务都执行完

  • 传入的回调函数执行时机：raf函数会在一次eventLoop的微任务之后执行，本次的渲染之前执行，是当前帧绘制之前执行。

  • 传入的回调函数中的参数rafTime为本帧开始的时间戳

  • requestAnimationFrame 的回调会在每一帧确定执行，属于高优先级任务

  • 页面在隐藏的时候，raf不会执行：document.hidden为true的情况下

requestAnimationFrame 比起 setTimeout、setInterval 的优势主要有两点：

  • requestAnimationFrame 会把每一帧中的所有 DOM 操作集中起来，在一次重绘或回流中就完成，并且重绘或回流的时间间隔紧紧跟随浏览器的刷新频率，一般来说，这个频率为每秒 60 帧。

  • 在隐藏或不可见的元素中，requestAnimationFrame 将不会进行重绘或回流，这当然就意味着更少的的 cpu，gpu 和内存使用量。

  • setTimeout，setInterval属于JS引擎，RAF属于GUI引擎

  • let height = null
setTimeout(() => height = 1.88, 0) //打印null
Promise.resolve().then(() => height = 1.99)// 打印1.99
requestAnimationFrame((time) => console.log(height))

    微任务可以拿到height的值，宏任务中拿不到height的值，所以说raf在js执行之后

定时器

对于网页上的定时器，来说settimeout有4ms的延迟，那么在做高精度的定时器，有误差。

原理

requestAnimationFrame() 方法接受一个回调函数，同时随机生成一个唯一的 handle 值作为标识符。回调函数和 handle 值共同组成一个元组 <handle, callback>，然后将这个元组推入动画帧请求回调函数队列中。

当页面可见时，如果动画帧请求回调函数队列中有元组，那么浏览器就会清空队列并且执行这些回调函数。

另外，每一个元组有一个 canceled 标识符，如果为 false，那么这个回调函数就不会在清空队列后被执行。cancleAnimationFrame() 方法的原理就是将这个标识符设置为 false。

requestIdleCallback函数

执行时机：有空闲时间才执行，属于低优先级任务。

缺陷：浏览器兼容不好的问题、FPS 20，50ms 刷新一次，远远低于页面流畅度的要求

MessageChannel -宏任务

属于 DOM event事件，优先级高于setTimeOut等计时器，所有先于他们执行，最高优先级的宏任务。该宏任务会在当前帧结束立刻执行。

使用raf函数，获得一帧（开始时间+16.6ms）传入管道中，MessageChannel响应事件，判断当前宏任务的开始时间（performance.now()）是否超过deadlineTime（），超过说明这一帧没有空余。

如同一个管道，是多个worker通信的方式

react中使用messageChannel和requestAnimationFrame 模拟了requestIdleCallback

react中为什么不用微任务做时间切片

​ 时间切片的目的是为了空出主线程让位给GUI线程，但是微任务的执行时机是必须要在产生自己的宏任务之前，渲染之前。如果使用微任务，会把任务都堆在渲染之前，造成js线程无法挂起，本末倒置。

​ 使用的宏任务，将未执行完的任务，推到下一帧执行。使用的是messageChannel和requestAnimationFrame 函数实现。

为什么不使用setTimeout呢

  1. 4ms的延迟

  2. settimeout有掉帧的情况，屏幕16.6ms才刷新一次，如果定时10ms，33.4秒的时候我们只看到了2次动画，但是属性可能已经变了3次

  3. settimeout是多少秒后推入队列，同步任务事件过长，也会让定时不准确

对setInterval的补充：
  1. Ajax 和 setInterval 没关系。
  2. 现代浏览器在页面失焦后都会限制 setTimeout/setInterval ，为了节省 CPU 资源。Reasons for delays longer than specified ，可以看这个解释。基本页面失焦后，每一次计时都会变成 1s。如果页面最小化了，这个时间间隔会更大。
  3. setTimeout和setTnterval都存在宏任务阻塞问题，虽然是在1000ms后推入队列，但是如何队列之前有大量宏任务，导致不准确。再次强调，定时器指定的时间间隔，表示的是何时将定时器的代码添加到消息队列，而不是何时执行代码。所以真正何时执行代码的时间是不能保证的，取决于何时被主线程的事件循环取到，并执行。
  4. setInterval 计时列队机制是：假如我设置时间间隔是10ms，setInterval 不是等上一个任务结束后等 10ms 再开始执行下一个，而是上一个任务开始后10ms 开始执行任务列队的下一个任务。如果上一个任务用了15ms 才执行完，下一个任务不会在第25ms 处执行，而是会在15ms 处立刻执行。
  5. 按照你的描述 requestAnimationFrame 是最好的解决方案。但 requestAnimationFrame 也有短板，这个方法的时间间隔是按照 屏幕刷新率 走的，默认是 60fps（60帧/s）。所以如果你的时间间隔要求小于 16ms，requestAnimation 是有点蛋疼的。而且 IE 10 不支持（现在还用 IE 的都应该拉出去打屁股）
实现倒计时的正确姿势：

使用时间戳，参数传入（Date.now() +30000(倒计时的毫秒数)），每隔一定的step，返回出当前的时间。

对于定时器的延迟问题，可以使用 worker + postMessage解决。初始化定时器阶段，实例一个worker，并且添加事件监听，在主页面调用的时候。存储时间，延迟，参数等等信息。在worker中使用setTimeout等计时器。
