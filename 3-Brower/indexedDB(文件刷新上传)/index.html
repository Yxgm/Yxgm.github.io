indexedDB（文件刷新上传）

需求背景，文件刷新续传，用户无操作

首先说明的一点：根据H5的标准，在没有用户的操作的前提下浏览器环境的js不可能去主动访问本地磁盘的文件的

1. 引导用户上传，storage存储偏移量(过滤已上传切片一个思想)

​ 没必要把文件内容存到浏览器storage里，你只需要根据你的每一片slice切片大小计算出每片在文件流当中的偏移量，即可在每次上传时按这个偏移量读取相应分片内容。

​ 设计好上传本身各个动作形成的这个状态机，你只需要在状态机里维护好当前上传到第几个分片即可，没必要记住分片内容，因为内容本身就存在本地磁盘上了。另外，上传的话最好先做md5校验，确认这个文件确实是你上次上传到一半的那个。

​ 刷新页面后，用户可以根据之前的操作状态提示用户重新选择文件，文件名提示一下这个文件是可以支持断点续传的，引导用户选中，然后按照刷新页面前最后一次的上传分片偏移量去做续传就好，上传完成后服务端校验分片重组后文件的完整性。

​ 在页面卸载的时候，window.addEventListener(“beforeunload”)事件，记录已经上传的文件切片索引数组，（在上传文件的时候，then方法中添加闭包变量hasUpload数组），因为切片上传是异步的，不一定按照发送的顺序，所以说需要记录所有已经上传的文件索引，我们先找到最后的切片索引值之后的文件进行切割，进行切片上传，同时索引前面的文件，可能有部分的切片漏发，所以说还要检验前面索引的连续性，拿到漏发的切片。最后进行切片文件发送。

2. IndexDB（非关系型数据库）

​ IndexedDB 做一个本地大文件持久化。切片上传记录上传状态，刷新后脚本提取 IndexedDB 中未完成上传的文件继续上传。完成则可以从IndexedDB 删除掉了。

如何使用

indexedDB的api很基础，所以说我们一搬封装一层（promise，并且配合ts使用效果会更好），同时需要注意对于数据objectStore的创建需要在onupgradeneeded回调中，否则会有bug。

add 和 put 的作用类似，区别在于 put 保存数据时，如果该数据的主键在数据库中已经有相同主键的时候，则会修改数据库中对应主键的对象，而使用 add 保存数据，如果该主键已经存在，则保存失败。

在索引上使用游标

同步和异步（Synchronous、asynchronous）

使用 IndexedDB 执行的操作是异步执行的，以免阻塞应用程序。IndexedDB 最初包括同步和异步 API。同步 API 仅用于 Web Workers，且已从规范中移除，因为尚不清晰是否需要。但如果 Web 开发人员有足够的需求，可以重新引入同步 API。

储存限制和回收标准

​ 浏览器的最大存储空间是动态的——它取决于您的硬盘大小。 全局限制为可用磁盘空间的 50％。 在 Firefox 中，一个名为 Quota Manager 的内部浏览器工具会跟踪每个源用尽的磁盘空间，并在必要时删除数据。

​ 因此，如果您的硬盘驱动器是 500GB，那么浏览器的总存储容量为 250GB。如果超过此范围，则会发起称为源回收的过程，删除整个源的数据，直到存储量再次低于限制。删除源数据没有只删一部分的说法——因为这样可能会导致不一致的问题。

3. 其他存储方式
File System API

​ File System API 和 FileWriter API 提供读取或写入文件到沙箱中（Sandboxed file system）。它是异步的，不推荐使用，因为 File System API 只能在 Chromium 内核中使用。

File System Access API

​ File System Access API 设计用于便捷得读取和编辑本地文件。但在读取或写入本地文件的时候，需要获得用户授权，并且授权状态无法持久化记录。

4. 数据存储localStorage放不下，indexDB不支持，如何做存储？（深信服2面）

​ webSQL早已经不在维护，并且只有少数浏览器（谷歌）支持，WebSQL更像是关系型数据库，使用SQL查询数据。pass！

​ cache Storage API，CacheStorage 接口表示 Cache 对象的存储。它提供了一个 ServiceWorker 、其它类型 worker 或者 window 范围内可以访问到的所有命名 cache 的主目录（它并不是一定要和 service workers 一起使用，即使它是在 service workers 规范中定义的），并维护一份字符串名称到相应 Cache 对象的映射。

​ cache Storage API 为缓存的 Request/Response 对象提供存储机制，常在 ServiceWorker 中应用。异步，不会阻塞主线程的执行，可以在 window、web workers、service workers 环境中使用。
