Vue版本区别

vue 3 性能提升大概1.3～2.0倍，vue3办变异。

vue 2 是面向对象开发，react 15.8 是函数式开发方案，vue 3 是半对象 半函数式开发方案。所以大厂全都开发用 react，函数式爽

1.预编译的优化

  1. 事件缓存：将事件缓存(如: @click)，可以理解为变成静态

  2. 静态提升：创建静态节点保存在render函数闭包中，之后的过程不用在走一遍h函数，后续直接复用，由于编译阶段的优化，除了能更快的生成虚拟 DOM 以外，还使得 Diff 时可以跳过”静态节点”

  3. slot插槽编译成懒加载

  4. 引出了block Tree概念，灵魂！

    openBlock函数会创建一个数组，在render函数过程中，会判断vnode是否是一个动态node（是否依赖一些动态数据，mastouch语法判断），是的话push到dunamic node数组中，这个数组是我们要更新的数据。之前的dunamicChildren和现在的dunamicChildren进行diff。也就是说静态节点连坐diff算法的机会都没有，效率提升

    Block tree 是一个将模版基于动态节点指令切割的嵌套区块，每个区块内部的节点结构是固定的。每个区块只需要追踪自身包含的动态节点。

  5. v-if和v-for的优先级调整，避免无用渲染。

2.响应式

vue2.0响应式的不足

  1. 功能限制，map，set class数据类型无法响应式，因为在进入observe的时候，判断当前为对象的时候，才会去劫持get和set。
  2. 可维护性：数组和对象实现不统一。
  3. 性能问题：深度递归，性能有所消耗。

一、 Proxy 代替 Object.defineProperty 重构了响应式系统，以下都是它的优势：

  1. 可直接监听数组类型的数据变化，因为数组也是对象

  2. 监听的目标为对象本身，不需要像Object.defineProperty一样遍历每个属性，有一定的性能提升，Object.defineProperty无法对新增属性进行监听，所以在初始化阶段，就必须声明所有变量，proxy代理的整个对象，所以可以新增属性，同时proxy还可以转发window的共享属性，请移步沙箱

  3. 可监听的副作用函数更加粒度化，effect函数

  4. 可拦截apply、ownKeys、has等13种方法，而Object.defineProperty不行

  5. 直接实现对象属性的新增/删除

  6. 在使用Proxy拦截对象的同时，搭配使用了Reflect去一些具体的操作

  7. 在get的时候 track收集依赖，set的时候trigger一下，执行更新函数，在delete的时候，也会trigger。

  8. trigger（target，key）函数的意义：更新数据和函数（渲染 watcher）的映射关系，track（target，key）函数意义：查找映射，找到函数，并且更新（可能是视图，也可能是函数在去间接的更新视图）。

  9. 针对可响应式对象每一个属性和它对应的更新视图函数使用weakMap来做存储，对于每个属性对应的watch，是用set来存储。目的就是为了在原来的可响应对象销毁之后，在weakMap中也会被垃圾回收，减少不必要的内存消耗。

二、vue3的响应式是惰性的

  • 在 Vue.js 2.x 中，对于一个深层属性嵌套的对象，要劫持它内部深层次的变化，就需要递归遍历这个对象，执行 Object.defineProperty 把每一层对象数据都变成响应式的，这无疑会有很大的性能消耗。所以说对于一个很深的对象，我们一般不会将它做成响应式，二是放在外面，或者使用Object.freeze()来处理对象。

    data数据具有数据层级结构，切勿过度扁平化或者嵌套层级过深，若是过度扁平化会导致数据命名空间冲突，参数传递和处理，若是层级嵌套过深也会导致vue数据劫持的时候递归层级过深，若是嵌套层级丧心病狂那种的，小心递归爆栈的问题。而且层级过深会导致数据操作和处理不便，获取数据做容错处理也比较繁琐。一般层级保持2-3层最好。

    同时如果vue中的data数据结构很深，数据将不会响应式，使用this.$forceUpdata(),强制更新。

  • 在 Vue.js 3.0 中，使用 Proxy API 并不能监听到对象内部深层次的属性变化，因此它的处理方式是在 getter 中去递归响应式，这样的好处是真正访问到的内部属性才会变成响应式，简单的可以说是按需实现响应式，就没有那么大的性能消耗。

  • 对于响应式对应两个api，ref和reactive，ref是reactive的加强版，ref可定义常量，也可对对象引用，vue3对开发者做出权衡，有人喜欢直接常量定义，有人喜欢用引用来定义。**ref对于原始数据类型是比reactive简单的，但是原理还是reactive({value: 10})**，但是对于复杂类型，ref的原理也是reactive，但是用ref相当于用reactive包装一下，用ref是可以，但是多了一步包装，所以对于复杂理性，建议用reactive，对于原始类型更建议用ref

3.打包体积变化

支持 Tree-Shaking，会在打包时去除一些无用代码，没有用到的模块，使得代码打包体积更小

4.diff算法

Vue2 是全量 Diff（双端对比），Vue3 是静态标记 + 非全量 Diff（最长递增子序列）

Diff 算法使用 最长递增子序列 优化了对比流程，使得虚拟 DOM 生成速度提升 200%，引入Bolck tree概念

diff算法详解 vue2、vue3、react的diff对比：https://juejin.cn/post/6919376064833667080#heading-2

5.生命周期

​ 新增了开发环境的两个钩子函数，在组件更新时 onRenderTracked 会跟踪组件里所有变量和方法的变化、每次触发渲染时 onRenderTriggered 会返回发生变化的新旧值，可以让我们进行有针对性调试

​ 删除最开始两个钩子（没有什么用），使用setup代替。

6.Ts的支持

​ vue2是支持类型的，用的是Facebook的Flow做类型检查，但是因为某些情况下推断有问题，所以改为支持ts。一个是为了更好的类型检查，另一个是拥抱ts

7.组合式API

​ composition api，终极逻辑复用，组织代码的解决方案，让代码更内聚。也是借鉴react hook，在使用的时候有心智负担，后来推出的script setup消除心智负担，其他拓展请看下文

vue3的心智负担

​ 因为CompositionAPI完全就是按照setup只会执行一次进行设计的。带来了许多心智负担，因为我们不得不注意什么情况下数据可以解构而且还要保持数据的响应式。

reactive包裹的数据，在setup中导出时，数据写成嵌套，将会导致响应式丢失。是因为setup只会执行一次，如下

迁移到新的心智模型

​ 时刻记住setup只会执行一次、永远不要直接使用简单类型、解构可能有风险，优先使用引用本身，而不是解构它、可以通过一些手段让解构变得安全

解决办法：

  1. 优先使用引用的本身，而不是解构它
  2. 使用computed包裹
  3. toRefs包裹。深层响应式 return { ...toRefs(data) }
  4. 使用JSX或h的时候，setup需要返回一个函数，这个函数其实就是render函数，它在数据变化时会重新执行，所以我们只需要把解构的逻辑放到render函数里，那么就解决了setup只会执行一次的问题。

建议：使用自定义hooks返回数据的时候，如果数据的层级比较简单，可以直接使用toRefs包裹；如果数据的层级比较复杂，建议使用computed。
