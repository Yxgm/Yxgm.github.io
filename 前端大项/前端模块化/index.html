前端模块化

1. 为什么要模块化

​ 对于刚开始 只是单纯的html css js ，后来随着前端的日益增长，在开发阶段我们需要更加细腻的划分模块，保证项目的可拓展，可维护性，最后再将模块全部打包起来，在合成一个文件，经历合-开-合，三个阶段。模块化主要是用来抽离公共代码，隔离作用域，避免变量冲突等。

2. 两种模块化方案

CommonJs：代码放在一个立即执行函数中，一些module，require函数依靠闭包存储，从而完成模块化。

ESModule: 语法层面的模块化。一般情况下我们写的 ES6 module 语法会还是会通过 bable 等工具转化为 CommonJS 语法的，原理：可见wbpack手写篇

3. CMJ和ESM的区别

ES-Module ：（import export方式）

特性 意义
1 只读引用 import的变量是只读的，不允许修改引入变量的值，当遇到import命令时，会生成一个只读引用。等到脚本真正执行时，再根据这个只读引用到被加载的那个模块里面去取值。如果修改引入变量会报错：SyntaxError: “count” is read-only。
2 动态、无缓存 对于动态来说，原始值发生变化，import加载的值也会发生变化，同时是是没有缓存的。
3 静态建立模块关系 模块关系的建立是发生在代码编译阶段，在编译阶段esm就要确定模块之间的依赖关系，导入导出必须位于模块的顶层作用域，而且必须是静态路径，在编译阶段就可以分析出模块的依赖关系。如果是动态路径将无法确认。
4 异步 ESM模块的import命令是异步加载，有一个独立的模块依赖的解析阶段。
5 循环引用 循环加载时，ESM模块仅仅是生成了引用，直到用的时候才会加载，并且没有缓存，每次都是动态的去被加载的模块中取值。所以esm根本不关心是否发生了“循环引用”，他只是生成了一个模块引用，需要开发者自己保证，真正读值的时候可以取得值。

Common-Js:（require，module.export）

特性 意义
1 浅拷贝 对于基本数据类型，属于复制，同时，在另一个模块可以对该模块输出的变量重新赋值。对于复杂数据类型，属于浅拷贝。由于两个模块引用的对象指向同一个内存空间，因此对该模块的值做修改时会影响另一个模块。
2 有缓存 使用require命令加载同一个模块时，不会再执行该模块，而是取到缓存之中的值，CommonJS模块无论加载多少次，都只会在第一次加载时运行一次，以后返回第一次运行的结果，除非手动清除系统缓存。
3 动态建立模块关系 模块依赖关系的建立是动态的，require的模块路径可以动态指定，支传表达式，if判断判断是否加载某个模块。因此，在cjs模块被执行前，并没有办法确定明确的依赖关系，模块的导入、导出发生在代码的运行阶段。
4 同步加载 CommonJS 模块的require是同步加载模块。
5 循环引用 循环加载时，脚本代码在require的时候，就会全部执行。一旦出现某个模块被”循环加载”，就只输出已经执行的部分，还未执行的部分不会输出。

加载原理（循环加载）参考链接

4. 总结：ES6 module 相对于 CommonJS 最大的区别就是两点：

  • 在执行模块前首先需要加载所有的依赖模块，如果加载有问题直接报错
  • ES6 module 的模块引入的都是变量，函数，类的引用这是很有先进性的，还有值得一提的就是可以按需引入自己需要的接口，方便tree-sharking

  2. export default 会往 exports 上添加 default 属性

4. Webpack模块打包

既然模块规范有这么多，那webpack是如何去解析不同的模块呢？

webpack根据webpack.config.js中的入口文件，在入口文件里识别模块依赖，不管这里的模块依赖是用CommonJS写的，还是ES6 Module规范写的，webpack会自动进行分析，并通过转换（不同的资源使用不同的loader去转化）、编译代码，打包成最终的文件。最终文件中的模块实现是基于webpack自己实现的webpack_require（es5代码），所以打包后的文件可以跑在浏览器上。

5. 模块化实现原理——IIFE（立即执行函数）

因为浏览器本身不支持模块化，那么webpack和require等模块化工具就用函数作用域来hack模块化的效果，整体打包生成的代码是一个IIFE（立即执行函数），函数参数是我们写的各个模块组成的数组，每一个我们说的模块，在这里就是一个函数。

1. Common.js实现思想

  1. 首先闭包变量存储installModules对象，定义__webapck__require函数，函数意思根据模块的id，来实现模块的require函数。__webapck__require函数会根据模块id，检查是否缓存了已加载的模块，有返回缓存exports
  2. 没有的话，在installModules，moduleId为key，模块为value，构造出一个模块对象，随后调用我们写的代码，会通过字符串拼接成一个函数声明，执行模块函数将module、module.exports和__webpack_require__作为参数传入。
  3. 注意这里做了一个动态绑定，将模块函数的调用对象绑定为module.exports，这是为了保证在模块中的this指向当前模块。
  4. 调用之后，标记模块为已加载，返回模块exports的内容
  5. 利用前面定义的__webpack_require__ 函数，require第0个模块，也就是入口模块。
  6. 就完成了对第一个模块的require，然后第一个模块会根据自己对其他模块的require，依次加载其他模块，最终形成一个依赖网状结构。webpack管理着这些模块的缓存，如果一个模块被require多次，那么只会有一次加载过程，而返回的是缓存的内容，这也是commonjs的规范。
  7. 所以说我们总是说模块，模块的实质是一个经过包装后的函数，exports，module，require作为参数的函数。

2. ES module实现思想

对比common规范打包，es Module下打包代码只有模版数组部分发生了变化，本身的iife函数结构是没有变化的。对于webpack是如何识别的，ast babel

  1. 首先通过Object.defineProperty在__webapck__require上添加属性__esModule，值为true，表明这是一个es模块，现在来说没什么用，之后会在两种模式混用的时候会发挥作用。
  2. 接着调用__webpack_require__函数，和common一样流程
  3. 最后在通过Object.defineProperty,将加在exports上的属性，设置属性描述符 configurable:false ,get使用取值器函数，保证是值的引用

Babel转化：直接export出去的变量，是加在export对象上的， default出去的是加在export的default属性上的，没读懂

1. babel 与 webpack 的关系
  • webpack：将 ES module、CommonJS、amd、cmd 等模块化通过自己内部的机制统一成 webpack 的模块化。
  • babel：转码 es6 语法，配合一些列 babel 工具链可以将新的 es2015+ 代码转换成所有浏览器支持的 es代码，babel 也可以将 es6 模块转换成 commonjs 模块
2. es module 与 commonjs 为何可以混用
  • 因为 babel 会把 es module 转换成 commonjs 规范的代码
  • babel 转码 es module 时如果遇到 export default 这种导出模块的书写方式后，会将其转换成 exports.default ，这时如果用 require 引入时，需要对其加上 .default 如 require('./a.js').default 这样才能获取 a 模块 export default 导出的 a
  • import 动态加载的模块也需要 .default 才能获取真实模块导出的值，如 import('./a.js').then(res => res.dafault)
